项目整体设计

1. 项目整体架构

项目的整体架构分为三层

应用层:调用服务层写好的API

服务层(flow svr):用于提供任务的相关接口,比如说任务创建,任务查询,查询待处理的任务,任务的治理并不单独作为服务,而是作为一个模块放在flowsvr中

worker层(执行层):通过flowsvr占据任务并且执行,执行后通过flowsvr更新任务的状态

任务治理:开线程去轮询做事情,比如说开一个线程去按照时间间隔定时查看表的数据有没有到500w,开一个线程查询是否有任务的执行时间太长的,如果有的话就把它重置,把它重置掉

项目是如何部署的?

多个flowsvr pod,部署到腾讯云集群上,通过clb(负载均衡访问)

多个worker pod,部署到腾讯云上

1个MySQL,1个Redis

2. 表结构的设计

任务信息表:t_${task_type}_task_${index},这里采用了一个分表的方式进行设计

task_type:任务类型,比如说是飞书机器人?那么表就定义为t_lark_task_1

index:第几张表?

  `id` bigint(20) NOT NULL AUTO_INCREMENT,-- 本条记录的唯一ID

  `schedule_log` varchar(4096) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '' COMMENT '调度信息记录',
  `priority` int(11) NOT NULL DEFAULT '0' COMMENT '优先级',
  `task_context` varchar(8192) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '' COMMENT '任务上下文，用户自定义',-- 用于存储json

关键字段解释:

  • max_retry_interval:最大重试间隔,设置了这个字段可以做阶梯重试,比如说设置为10,当任务失败了之后,这时候就需要距离上一次任务的上一次执行
  • schedule_log:调度的日志,包括这个任务本次调度的trace id以及耗时,可以设计成以下的格式

任务配置表

CREATE TABLE `t_schedule_cfg` (
  `task_type` varchar(128) COLLATE utf8mb4_unicode_ci NOT NULL COMMENT '任务类型',
  `schedule_limit` int(11) DEFAULT '0' COMMENT '一次拉取多少个任务',
  `schedule_interval` int(11) NOT NULL DEFAULT '10',
  `max_processing_time` int(11) DEFAULT '0' COMMENT '处于执行中的最大时间',
  `max_retry_num` int(11) DEFAULT '0' COMMENT '最大重试次数',
  `retry_interval` int(11) DEFAULT NULL COMMENT '重试间隔',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP,
  `modify_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`task_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci

task_type:任务的资质,服务端要根据任务的资质执行对应的处理逻辑,比如说task_type是音视频,那么就要在配置表中插入这个任务的一些基本配置

max_processing_time:用于配置这一类资质任务的最大执行时间,当超过了这个时间就执行一个扫描的操作

retry_interval:某一类任务的重试间隔,当某个具体的任务没有指定重试间隔的时候,这时候就采用配置表中的重试间隔

schedule_limit:worker一次拉取多少个任务来执行,相当于limit字段后面接入的数字

select * from t_lark_task_1 limit 0,schedule_limit
insert into `t_schedule_cfg`(task_type, schedule_limit, schedule_interval, max_processing_time,
max_retry_num, retry_interval) values('lark', 100, 10, 30, 3, 10);

表示lark这个任务,任意一个worker每10s去调度一次任务,每次最多拉取100个任务来执行,最大处理时延是30s,每次失败之后等待10s后再恢复调度

3. 状态的流转

这里主要描述了任务的流转状态,一个任务的生命状态是:

  • 等待中:代表用户提交了任务,但是还没有被worker占据并且执行操作
  • 执行中:代表任务被底层的worker占据了
  • 失败:任务执行失败
  • 成功:任务执行成功

一个任务在执行过程中,可能有两种情况:

第一种情况,当本阶段任务执行成功了,还没有彻底完成(意思说还在执行阶段),这时候又会返回任务池,等待被worker占据

第二种情况,当本阶段任务执行失败了,但是还没有达到最大的重试次数,这时候又会返回任务池,等待被worker占据

第三种情况,当本阶段任务执行失败了,达到了最大的重试次数,这时候就会直接宣告任务失败,上报业务层

第四种情况,当本阶段任务执行成功了,任务所有阶段都完成了,这时候宣告任务成功

4. flow server设计

flow server是什么?首先先来回答它的定义,我们才能了解我们如何去设计它

flow server可以理解成一个代理客户端,它对内作为一个网关,将外部的请求在内部路由到具体的执行函数,最重要的是它提供了一个入口给worker来拉取服务,其中最核心的功能就是

  • 创建任务,当前台发起一个调用的时候,会先去调用这个方法
  • 占据一批任务,底层的这个worker通过这个接口来占据任务池中的任务
  • 获取任务列表,提供了关系化管理任务的能力
  • 更新任务信息
  • 获取任务的配置列表

以下是在Java中的实现

Task实体类

@Data
@ToString
@AllArgsConstructor
@NoArgsConstructor
public class Task {
    /**
     * 任务的ID
     */
    private String taskId;
    /**
     * 任务类型
     */
    private String taskType;
    /**
     * 任务的执行到哪个阶段了
     */
    private String taskStage;
    /**
     * 任务执行上下文,通过此上下文可以复现任务在执行过程中场景
     */

    /**
     * 调度日志
     */
    private String scheduleLog;
    /**
     * 调度优先级
     */
    private Integer priority;
    /**
     * 创建任务的用户ID
     */
    private String userId;
    /**
     * 任务的状态
     */
    private Integer status;
    /**
     * 任务目前已经重试的次数
     */
    private Integer certRetryNum;
    /**
     * 该种类型的任务最大重试的次数,当超过了这个次数就要将其设置为失败了

     * 该种类型的任务的最大重试间隔
     */

flowserver接口设计

public interface FlowServerService {

    /**
     * 创建一个任务
     * @param taskData 任务的元信息
     * @return 返回一个任务的ID
     */
    String createTask(Task taskData);

    /**
     * worker调用此接口去占据一批任务
     * @param taskType 占据的任务类型
     * @param limit 一次最多占据多少个任务?要求不超过2000条数据
     * @return 返回占据的任务列表信息
     */
    List<Task> holdTask(String taskType, Integer limit);

    /**
     * 后台管理面板中查看任务列表
     * @param taskType 查看的任务类型
     * @param status 指明要查询哪种任务的状态
     * @param limit 一次查询多少条,不超过2k
     * @return 返回查询的列表
     */
    List<Task> getTaskList(String taskType,Integer status,Integer limit);

    /**
     * 更新任务的基本字段信息
     * @param taskId 任务的ID,通过此ID定位到任务然后执行
     * @param status 任务的状态
     * @param context 任务上下文
     * @param stage 任务的执行阶段,这里规定,如果为""的话,那么就不更新
     * @param scheduleLog 调度日志
     * @return 返回更新是否成功?
     */
    Boolean updateTask(String taskId,Integer status,String context,String stage,String scheduleLog);

    /**
     * 根据输入的信息查询任务配置表
     * @param taskScheduleCfg 输入的查询信息
     * @return 查询任务配置结果
     */
    List<TaskScheduleCfg> getTaskScheuleCfgList(TaskScheduleCfg taskScheduleCfg);
}

5. worker设计

worker的定义:按照flow server提供的信息执行任务的真正底层组件,它被分布式地部署,是这样的话就可充分利用存量机器的性能来执行任务,worker通过hold_tasks接口来占据任务,由于worker是分布式部署的,因此这多个worker获取任务的时候就可能会导致一个并发的问题

并发问题(多机竞争)的解决方案是基于分布式锁来实现的
