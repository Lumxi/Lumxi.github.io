项目关键设计剖析

1. 多机器/多进程竞争问题

问题产生的背景

在部署了多个worker的时候,这时候可能会有多个worker同时向flow server发起hold task请求,而flow server请求的都是同一个数据库,因此在这样的情况下,就可能导致多个worker占据的同一个任务这样的问题,就会导致同一个任务被执行多次,这种问题的解决办法有很多,可以将解决方案放到业务侧去解决,解决幂等性,防止重复执行,但是这个问题的根本原因是asyncflow没有做并发控制,所以在这样的情况下,就要解决这样的多机器竞争的问题

那么从框架上来看,如何来解决这个问题呢?

  • 第一个想法,通过加锁来排队,这样的话可以保证同一批任务可以被有序拿到
  • 第二个想法,通过架构的优化来解耦

第一种方案,通过MySQL的行级锁来实现加锁

这里的话由于是数据库层面的锁,因此需要分析加锁的情况

-- 取任务
select task_id from t_lark_task_1 where status = 1 limit 100 for update;
-- 更新任务
update t_lark_task_1 set status = 1 where task_id="taskid4"

假设这是两个事务并行操作,第一个事务是worker在占据任务,第二个事务是worker在执行任务成功后,将任务返回任务池的这个操作

那么如果是采用数据库加锁的这个方案的话,就会导致出现两个完全不相干的事务因为加锁的问题而阻塞,这样的话

具体分析

首先select的查询条件是status,因此在这样的情况下,本质就可以抽象成一个非唯一索引了的查询了

非唯一索引查询分析:

由于查询条件是status = 1,因此[t1,t2,t3]都会被上一个next-key lock,而[t4]作为第一个不符合条件的记录,会上一个间隙锁,也就是将(t3,t4)这个范围内的插入给阻塞掉,那么update是否会造成影响呢?

首先我们要理解update的操作在底层实际上是delete && insert

因此delete的话是不会造成阻塞的,但是insert的话可能会造成阻塞,一旦你插入的这个task_id刚好落在间隙锁的区间的时候,就会造成阻塞,这是个不确定的因素

影响:降低性能,同时导致原本程序中互不相关的事务操作产生干扰,同时因为数据库操作频繁,容易造成死锁,数据库治理难度大

第一种方案是用了一个数据库级别的悲观锁,主要问题在于锁不可控,不同业务属性的事务之间相互干扰,可能造成死锁

因此这里的话提出第二种方案:使用一个乐观锁的机制

其实说白了这个方式就是加个字段,owner,通过owner字段是否为空或者是否为自己来确定任务是否被占据好了

-- 查询任务池
select * from t_lark_task_1 where status = 1 order by order_time limit 2;
-- 将这批任务占据下来
update t_lark_task_1 set owner = a and status = 2 where task_id in (task1,task0) and status = 1
-- 查询真正被占据的任务
select * from t_lark_task_1 where status = 2 and owner = a

这种方式可以避免直接加行级锁导致的锁不可控的问题,但是也有一定的缺点:

  • SQL语句太多,可能导致长事务
  • Worker越多,冲突越大,写越多
  • 可能导致很多的请求都是无效的

这种方案本质上还是在数据库层面控制并发,因此实际上性能还是比较低下的

第三种方案,基于分布式锁解决多机器竞争的问题

分析上面的问题,归根到底就是有同时多个worker去请求同一个flowsvr的holdTask接口,一个直接的思路就是,一个flowsvr的holdTask的接口在一个时间段内只能够被一个worker所调用,由于这个锁是跨机器的,因此这时候就要用到分布式锁了

使用分布式锁的核心就是在请求holdTask必须获取到锁之后才能执行,但是这种方案也有一个问题:

  • 当产生冲突的时候,Worker闲置,得不到利用,同时CPU空转
  • 锁释放不及时,其他Worker限制

回顾原因,为什么冲突的时候Worker会闲置?是因为都没有任务,一个核心逻辑是:只有获取到了任务了才能够执行任务,这个逻辑是串行执行的。

第四种方案,将拉取任务和执行任务解耦,拉取任务和执行任务不一定串行执行,而是有可能异步执行的

这里的话就需要引入一个同步数据的服务,由这个服务来定时地将数据库中的未执行任务拉取下来,然后投放到Kafka中,然后worker订阅了Kafka之后,就能够执行了任务了,这样的话就将拉取和执行解耦了

优点:

  • 架构清晰,解耦
  • 压力稳定,支持水平扩容

缺点:

  • 需要一定的容灾成本,比如说Kakfa挂了怎么办?同步服务挂了怎么办?
  • 执行链路变长了,监控数据量变多
  • 引入了消息队列,如果消息队列不可靠就可能导致服务整体不可靠

2. 最大重试间隔

通常来说,一个任务的重试策略可以分为是均匀式重试和渐进式重试,在本框架中是基于interval这个字段来实现的

比如说-10,表示说这个任务是均匀重试的,也就是每隔10s就重试一次

比如说10,表示说这个任务是渐进式重试的,也就是[1 2 4 6 8 10]这样的间隔来重试

更加丰富的重试策略有:

  • 按照用户的自定义策略来进行重试,比如说字符串1,2,4,8,10直到最大的重试次数
  • 通过Lua脚本来实现,也就是说将重试时间写成Lua脚本,去解析这个脚本

3. 优先级设计策略

所谓优先级策略设计,就是当同步服务去拉取服务的时候,要优先拉取哪些任务

CREATE TABLE `t_lark_task_1` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `user_id` varchar(256) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `task_id` varchar(256) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `task_type` varchar(128) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `task_stage` varchar(128) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `status` tinyint(3) unsigned NOT NULL DEFAULT '0',
  `crt_retry_num` int(11) NOT NULL DEFAULT '0' COMMENT '已经重试几次了',
  `max_retry_num` int(11) NOT NULL DEFAULT '0' COMMENT '最大能重试几次',
  `max_retry_interval` int(11) NOT NULL DEFAULT '0' COMMENT '最大重试间隔', 
  `schedule_log` varchar(4096) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '' COMMENT '调度信息记录',
  `priority` int(11) NOT NULL DEFAULT '0' COMMENT '优先级',
  `task_context` varchar(8192) COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '' COMMENT '任务上下文，用户自定义',
  `order_time` int(20) NOT NULL DEFAULT '0' COMMENT '调度时间，越小调度越优先',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `modify_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_task_id` (`task_id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_status` (`status`),
  KEY `idx_tasktype_status_modify_time` (`status`,`order_time`)
) ENGINE=InnoDB AUTO_INCREMENT=5 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci

回顾数据库字段,在这里可以看到能用来做优先级的依据有:priority(优先级)、order_time(调度时间),调度的话可以写出这样的代码

select * from t_lark_task_1 where status = 1 order by priority desc,order_time limit 100;

核心就是将priority加入排序,这样的高频操作实际上是需要建立索引来提高性能的

但是缺点有:

  • 需要将priority加入联合索引,影响性能(因为索引键的长度变长了,在维护的时候需要额外的开销),并且绝大多数priority都是一样的,开销大
  • priority是固定的,不够灵活,比如说优先度高的任务,当失败了之后,又会马上被调度,对于那些优先级低的任务容易导致一个饥饿的现象

第二个方案,观察现有的索引

PRIMARY KEY (`id`),
UNIQUE KEY `idx_task_id` (`task_id`),
KEY `idx_user_id` (`user_id`),
KEY `idx_status` (`status`),
KEY `idx_tasktype_status_modify_time` (`status`,`order_time`)

能不能不用priority这个字段呢?

order_time抽象为一个排序字段,可以包含有创建时间,更新时间,重试间隔的影响,这里的话可以再给它一个语义

表示优先几秒,最大优先一年(足够大)

创建的时候:order_time = 当前时间-priority,表示说优先priority排序,比如说priority=10的时候,相当于是提前10s创建的任务

调度成功之后:order_time = 当前时间- priority,始终维持一个优先的秒数

调度失败之后:order_time = 当前时间+算出来的重试间隔,表示说要等一会再调度,可以避免饥饿

这种方案比较灵活,综合考虑创建任务,调度任务成功(需要保持下一次调度的优先级),调度任务失败(需要保证隔一段时间再调度)的这些情况

问题:这个项目调度任务的优先级是如何设计的?

当时做这个设计的时候,设计了两个方案,第一个方案是直接按照用户配置的priority进行排序,但是这样做几个缺点,首先第一个就是索引的开销增大了,因为索引字段增长了,影响了性能,第二个就是不灵活,priority是用户配置死的,在任务调度过程中,可能会成功,失败,为了避免任务长久得不到执行,这时候就需要动态更新priority了,在后面的做法是将priority秒化,不表示级别,而是说它应该要优先多少秒执行,将priority的信息融合到order_time中,这种方案耦合度小,成本低,足够灵活

调度任务的入口在holdTask中,其中的SQL语句是select * from t_lark_task_1 where status = 1 order by order_time desc limit 100这样的语句,可以看到这句用到的索引是status和order_time,其中status是查询条件,而order_time则是用来排序的字段,这里的话对order_time做了一个语义的规划

当创建任务的时候,这时候order_time = 当前时间 - 优先级,这是为了让用户规定的优先级起作用

当调度任务成功之后,这时候保持order_time = 当前时间 - 优先级,这是为了下一次任务能够按照正常的节奏被调度

当调度任务失败之后,这时候更新order_time = 当前时间 + 重试间隔,当任务失败了之后,就能够隔一段时间再调度

4. 多阶段任务

多阶段任务是我们这个框架的一个基本场景之一,下面来了解一下

多阶段任务,阶段之间最好是同步的,第二阶段需要等待第一阶段完成之后才开始,比如说视频处理

  • 审核(鉴黄异步)
  • 转码(异步操作)

中间可能还有获取元信息的操作

多阶段的任务task_stage来进行描述的,最简单的就是一个switch-case的方式,找到task_stage匹配的分支

阶段流转操作

阶段流转:指的是任务在执行成功后,流转回到任务池的这个过程,由于多阶段任务存在一个同步执行的场景,因此在这样的情况下,就必须要采取一个手段,将上一阶段执行完的结果持久化起来,这里的方案是将产生的结果通过json序列化的方式存储到mysql的字段中去

同时更新任务阶段,当流转回到任务池的时候,下一阶段再被执行的时候就会执行新的函数

任务处理回调如何接入?

  • 上下文存储:支持上下文存储来保存任务执行过程中的阶段信息
  • 支持阶段的跨越,可以阶段间顺序流转,也可以根据用户定义来灵活跳转到任意阶段
  • 自动重试:流转失败不需要额外操作,返回错误的时候框架会自动重试,直到达到最大的上限

5. 数据库分库分表

回顾一个基本的表的结构

CREATE TABLE `t_schedule_pos` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `task_type` varchar(256) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci NOT NULL DEFAULT '',
  `schedule_begin_pos` int(11) NOT NULL DEFAULT '0' COMMENT '调度开始于几号表',
  `schedule_end_pos` int(11) NOT NULL DEFAULT '0' COMMENT '调度结束于几号表',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `modify_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_task_type` (`task_type`)
) ENGINE=InnoDB AUTO_INCREMENT=1 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci

其中关键的字段schedule_begin_pos和schedule_end_post表示实际存储的表后缀从[begin,end]开始,一开始都为1,表示只有一张表

分表的流程

  1. 当1号表的数量达到了阈值
  2. 任务治理模块通过扫描,发现1号表记录数量达到了阈值,触发一个分表的操作,也就是去创建2号表
  3. 经过步骤2之后,此时的状态是占据任务请求走1号表,也就是schedule_begin_pos为1,但是新创建的任务的请求是走2号表的,也就是schedule_pos_end为2
  4. worker持续占据任务,消费任务,直到1号表被消费完毕
  5. 此时worker去占据任务的时候,如果发现1号表中都没任务了,就会将schedule_begin_pos变成2,此后,占据任务请求和写入任务请求就都是到2号表了

接下来从数据库的角度来剖析这个问题:

  1. 在初始状态的时候,shcedule_begin_pos和schedule_end_pos都是1,此时占据请求和消费请求都是走1号表
  2. 1号表记录达到一个阈值,并且创建2号表的时之后,这时候由于schedule_begin_pos=1的表中依然还有任务没有被消费完成,所以在这样的情况下,此时占据请求依然会走到1号表上,此时的情形就是占据请求走1号表,写入请求走2号表
  3. 消费完1号表之后,shcedule_begin_pos就会变成2,此时占据请求和写入请求都会走2号表

一般来说,检查一个表中是否任务全部被消费完毕,可以用这个SQL

select count(*) from t_lark_task_1 where status not in(4,5)
-- 不是成功,不是失败,肯定就是要待执行的了

常见的问题:

  • 介绍一下项目中的分表方式是怎么样的?

项目的分表是数据驱动动态分表的,也就是说表的数量会随着记录的增加而增加,具体的手段是:

当任务数量达到了一定的阈值之后就会发生分表,分表之后新增的任务就会写入新表,而当旧表中的任务被消费完毕之后,才会去走新表,消费几号表,写入几号表,这个信息也是通过一张单独的任务位置表来实现的

  • 为什么要按照大小分表?

这是因为任务执行完毕了之后就处于一个冷数据的状态,查询的频次实际上是很少的,但是由于框架支持关系化查询的能力,所以在这样的情况下,就希望保留表,但是如果表太大,那么在后续拉取任务的时候很容易产生性能瓶颈,所以我们的分表策略是冷热分表。

分表之后,任务的拉取和任务的消费都是集中在一张数据规模比较小的表中,可以提高性能

同时针对关系化的能力设计,我们可以在id字段中添加一个后缀,比如说task_id_1,这样就知道要查询这个任务,就要到1号表中去查询了,而不需要扫描全部表

  • 实际场景量很少,但是你的分表量是500w?

首先这个作为一个框架,需要能够应对不同的情况,提供分表能够让框架不那么玩具化,同时对于一个异步处理的框架,一些任务动辄几千万的数据量也是完全有可能的,所以抽象出来做一个框架,最好把这些都考虑进去想

这个算是一个前瞻性设计而不是一个过度设计

  • 读写请求大到单个Mysql数据表处理不过来怎么办?

首先要拉齐和面试官之间的认知,他到底想问啥?应该是说是请求量很大,然后TPS太高,MySQL扛不住。

一般来说,这样的问题就是要考虑一个水平扩展,这个水平扩展简单来说就是做一个负载均衡,让MySQL的请求均匀地打到提前部署好的数据表中,如果单表确实性能不够了，我们也可以支持按用户Hash分片，一个比较容易的方式是接入tdsql这种完全兼容mysql的数据库，可以自动分片。另外我们也打算提供mongodb这样天然支持分片的文档数据库用作任务管理，这个可以根据场景选择的。

  • 会不会出现schedule_beign_pos和schedule_end_pos跨不止1张表的情况？

会，但可能性不高。比如突然一波海量流量，比如worker挂掉，长时间积累。

这种情况要么等待逐渐消耗，要么需要增加worker来提升吞吐。

6. 连接池设计

什么是MySQL连接池?

数据库连接池(Connection poll)是程序启动的时候建立足够多的数据库连接,并将这个连接组织成一个连接池,由程序动态地对池中的连接进行申请,使用,释放

为什么需要MySQL连接池?

创建数据库连接是一个非常耗时的操作,底层包括有TCP的三次握手和四次挥手,同时也会消耗掉一定的连接报文流量,同时MySQL维护连接的手段是通过线程来维护的,也就意味着线程:连接是1:1的关系,这样的话大量的系统资源都被浪费在上下文切换上,最终导致系统资源被无谓的浪费掉

解决方案

当有连接池的时候,可以复用连接,比如说100个请求,那么一定需要100个连接吗?不一定,只有在发生网络交互的时候需要用到连接对象,因此在这样的情况下,当SQL处理完毕之后,进入一个底层系统的渲染的时候,不涉及到连接对象的时候,就可以把之前建立好的连接对象拿出来使用了

适合场景

连接池仅在超大型应用中才有价值，连接池并没有提高sql的查询速度，连接池是用来保护数据库的,限制连接数,为了避免连接过多导致数据库崩溃，不是用来提升性能的

建立连接—-执行业务逻辑—–销毁

长连接性能是优于短连接,不需要重复的去建立连接,常驻内存要复用连接

普通的应用采用MySQL长连接方案，每个worker创建一个MySQL连接，每台机器开启100个worker进程。如果有10台机器，每台机器并发的请求为100。实际上只需要创建1000个MySQL连接就能满足需求，数据库的压力并不大。即使有100台机器，硬件配置好的存储服务器依然可以承受。

达到数百或者数千台应用服务器时，MySQL服务器就需要维持十万级的连接。这时数据库的压力就会非常大了。连接池技术就可以派上用场了，可以大大降低数据库连接数,减入服务器压力。

MySQL连接池创建机制

数据的连接池为维持若干个长连接，当新请求到达的时候，如果连接池有空闲连接，就分配给连接池去处理，否则，后面的数据库连接请求将被加入到等待队列中。

数据库连接池的机制：

  • （1）程序初始化时创建连接池
  • （2）使用时向连接池申请可用连接
  • （3）使用完毕，将连接返还给连接池
  • （4）程序退出时，断开连接，并释放资源

1、启动时创建N个MySQL连接，收到客户端发来的SQL后，分配1个MySQL连接，将SQL发往数据库服务器。然后等待数据库返回查询结果。当数据库返回结果后，再发给对应的客户端连接。

2、核心的数据结构是3个PHP队列结构。idle_pool是空闲的数据库连接，当有SQL请求时,从idle_pool中移到worker_pool中。当数据库返回结果后从worker_pool中再移到idle_pool中，以供新的请求使用。当SQL请求到达时如果没有空闲的数据库连接，那会自动加入到wait_queue中。一旦有SQL完成操作，将自动从wait_queue中取出等待的请求进行处理。

mysql客户端连接池，是指向mysql发起请求的一方缓存住连接，反复使用的技术。注意是在MySQL的客户端，在我们的场景，就是flowsvr。

mysql最大连接数是服务端参数，表示最多能开多少个连接，超过这个连接就会报too many connnections，这个值在我们的场景设置为几千比较好。
