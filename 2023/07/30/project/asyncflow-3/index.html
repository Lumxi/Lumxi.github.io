项目组件的具体实现

1. 创建任务的实现

创建任务这个过程可能是由前端发起,也可能是后端服务器发起RPC调用实现的

  • flow server收到http包之后,路由到handle函数,在Java中,这一个由SpringMVC帮我们实现了
  • 检查参数是否合理
  • 执行任务,具体是找到要插入几号表?向数据库中插入任务的元数据信息,填充返回包
  • 依赖框架给用户回包

这里来理解一下相关操作的意图,拿到任务配置表信息是为了填充任务的元数据信息,当然这里也支持用户自定义该任务重试间隔等基本属性

拿到插入的表号是为了知道当前这个记录可以插入到哪个表中,清楚了这个,来看看代码是如何实现的

@Override
public String createTask(Task taskData) {
    log.info("开始执行创建任务的需求,任务信息:{}",taskData.toString());
    String taskId = "";
    try{
        // 1. 获取任务配置表的信息
        TaskScheduleCfg taskScheduleCfg = taskScheduleCfgMapper.selectByTaskType(taskData.getTaskType());
        // 2. 计算出当前记录要插入到哪张表中?
        taskId = TaskDataUtil.generateTaskId();
        // todo:这里的话就先将全部插入到1号表中,之后再做分表的具体操作
        String tableName = TaskDataUtil.getTableName(taskId,1, 1, taskScheduleCfg.getTaskType());
        // 3. 填充数据
        this.fillTask(taskData,taskScheduleCfg);
        // 4. 插入数据库记录
        taskMapper.insertTask(tableName,taskData);
    }catch (Exception e){
        e.printStackTrace();
    }
    // 5. 返回taskId
    return taskId;
}

private void fillTask(Task taskData,TaskScheduleCfg scheduleCfg){
    taskData.setCertRetryNum(0);
    taskData.setMaxRetryNum(scheduleCfg.getMaxRetryNum());
    taskData.setRetryInterval(scheduleCfg.getMaxRetryInterval());
    taskData.setCreateTime(LocalDateTime.now());
    taskData.setModifyTime(LocalDateTime.now());
}

大致的思路是这样的,具体的实现上还有得改进,目前是以理解原理为主

2. 占据任务(flowsvr侧实现)

这个流程由worker触发,请求到flowsvr

  • worker发起占据任务的请求,可能是通过http调用的方式来请求的
  • flowsvr接收到了之后,会根据路由找到占据任务的函数
  • 检查参数
  • 执行请求,具体来说是要找到要拉取几号表(目前没有实现分表)从数据库中拉取一批任务,然后将这批任务设置为执行中,填充一个返回包

@Override
public List<Task> holdTask(String taskType, Integer limit) {
    limit = checkLimit(limit);
    List<Task> result = new ArrayList<>();
    try{
        // 1. 找到要拉取的表
        // todo:此处没有做分表
        String tableName = TaskDataUtil.getTableName(1,1,taskType);
        // 2. 从表中拉取一批任务,将这批任务设置为执行中
        // 拉取规则:正在等待执行 && 按照创建的时间
        List<Task> tasks = taskMapper.selectByTaskType(tableName, limit);
        // 设置为执行中
        if(tasks == null || tasks.isEmpty()){
            return result;
        }
        // 收集List
        List<String> ids = new ArrayList<>();
        for (Task task : tasks) {
            ids.add(task.getTaskId());
        }
        Integer affectedRows = taskMapper.updateTask(ids, tableName);
        if(affectedRows != null){
            log.info("[service]holdTask修改的数据行数为:{}",affectedRows);
        }
        // 3. 返回任务详情
        result = tasks;
    }catch (Exception e){
        e.printStackTrace();
    }
    return result;
}

3. 任务调度流程

这个是最复杂的函数,也可以说是整个系统最为核心的功能,这里的话先剖析一下其中一些比较关键的点

第一步,先去调用flow server的holdTask接口,这时候就会得到一批数据,这批数据就是将要处理的数据

第二步,当得到了这批数据之后,载入上下文信息,根据上下文信息去执行任务

第三步,拿到任务执行结果,更新本地任务信息

第四步,执行任务收尾工作

第五步,调用flow server去更新远端的任务信息

整体来说就是这个流程
