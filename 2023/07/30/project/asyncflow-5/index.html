项目题库与问题

1. 介绍一下项目,为什么做这个框架?什么场景?

介绍这个项目先从我们的一个业务背景说起吧,那是在电信子公司实习的时候,那时候接到一个需求和mentor一起开发一个异步报表导出的功能,这个需求的背景是当前数据库一些数据的存量到达了1kw,如果按照传统的报表导出的逻辑,是这样的:

前端发起一个请求,后端收到请求后,在后端完成原始数据的加工(包括有数据生成,数据脱敏,数据加密)、文件生成、文件上传到OSS中,后端再返回一个链接到前端,如果数据量很少,那么连接很快就能够响应过来,如果数据量很大,那么极有可能在处理数据这一步就会超时,占用HTTP连接,而且给用户的观感就是,一直在这转圈圈,然后系统就用不了了

所以在这样的情况下,就需要一个异步报表导出的能力,我们之前是前端发起请求之后,就一直在等链接返回嘛,那么这里的话把它异步化了,当前端发起请求之后,后端在一张事件表中写入一条记录,表示xx报表导出事件,这条记录写入成功之后,就会返回了,前端不必等待链接生成返回。

后台会自动地扫描这张任务表,扫描到了这张任务表之后就会去执行任务

但是会有一个问题:任务失败了怎么办?

失败的场景通常是在任务在执行过程中,有人执行服务器更新之类或者服务器宕机之类的,或者是调用第三方接口请求数据的时候导致一个失败的问题

通常来说比较难受的一个情况是这样的:一般来说,生成一个报表最为耗时的阶段是执行数据计算的阶段,如果数据计算完毕了,然后没有把数据写入到报表中,是比较浪费的,但是对任务的设计是只有待执行、成功、失败三种状态,没有一个中间的状态,如果有一个中间的状态,可以根据这个中间的状态去执行一个续做的时候,能够节省掉一些资源

由于工期比较紧,于是仅做了第一版的建设,就是能够完成异步化能力建设就结束了,后面的话结合我自己的思考以及第一期实现的相关痛点,抽象出来了这个框架

痛点:

  • 多阶段任务执行过程中,不能保存中间结果,计算资源是比较珍贵的,如果每一次的重试都是重头来过,会比较浪费资源**(资源浪费–保存执行上下文)**
  • 不能做到自动重试和执行任务的优先级设计不合理,可能导致任务饥饿得不到执行的问题**(任务重试和任务优先级设计不合理–设计重试字段和优先级字段)**
  • 没有任务治理:对于这张任务表而言,其实只有那些还没有被扫描到的任务才有执行的必要,而执行任务执行扫描的时候,是整个表都去查询,有很多无用的开销**(开发任务治理模块,冷热分表)**

基于这些痛点,抽象出来了AutoJobs框架,这个框架的主要就是用来完成任务的自动执行与生命周期管理,项目架构主要可以分为两层服务层和执行层

其中服务层JobsServer负责创建任务以及相关的任务管理API,可以看做是一个网关

而执行层Worker则是负责去读取JobsServer创建的任务,并且读取任务,可以看做是底层实际执行操作的线程

2. AutoJobs和消息队列的区别是什么?

这个问题可以从这两者之间的定位说起,AutoJobs的定位是一个框架,它主要的用途是用来接入一种类型的任务,然后接入完成后,每当有新任务注册到框架中,这个框架就可以自动地去执行这些任务,同时还提供了容错、数据看板,任务治理的能力

消息队列的定位是消息引擎,本质上是消息中间件,各个服务之间通过这个中间件来完成异步化的通信,同时也可以通过消息队列的来完成削峰填谷,也就是完成上下游服务的生产效率和消费效率不对等的问题

总的来说,AutoJobs是一个任务调度框架,它主要负责任务的生命周期管理,自动执行,专注于任务,而消息队列用途广泛,本质上是一个中间件,充当异步化解耦,削峰填谷的角色,更加通用。

所谓容错,就是在执行任务过程中失败的话,能够自动重试,当运行框架服务宕机之后,再次拉取任务,能够根据执行过程中的一个上下文环境复现任务执行的流程,这样的话就实现了一个自动续做的功能

所谓数据看板,指的就是任务关系化能力的建设,我们希望能够提供一个接口,返回框架接入后的任务执行情况,因此要对任务数据进行持久化,哪怕它们不会再被底层的Worker组件拉取到,这个能力主要是为了排查一些异常情况,比如说有些任务执行了最大重试次数都还是失败,这时候就需要人工介入了,

所谓任务治理,指的就是冷热数据的治理,一个框架,可能接入很多任务,一天可能执行上w个任务,如果放任这样去增长,最严重的问题就是任务执行效率的低下,因为我们拉取任务最终还是到表中去读取的,如果表中数据太多,那么就会导致的任务的读取流程会很慢,那么任务治理就是说,尽可能地分离冷热数据,当拉取数据的时候只读取热数据,提高拉取任务的效率,同时对于一些执行失败的任务,通过打标的方式,来控制这个任务下一次执行的间隔,既可以避免任务无意义的重复,也可以避免其他任务饥饿

到这里可以会被问:这个框架能用消息队列替代吗?

到这里的话可以稍微扯扯皮,说看需求,如果任务的执行不需要类似于一个数据看板的功能的话,只关注结果的话是可以不用这个框架,用消息队列的消息流转模型实际上也可以实现这个功能,消息流转的模型就类似于多个信箱,每个信箱都存储单一阶段的任务.

3. 业界已有的异步框架调研过吗?为什么不用?

这个问题先说自己调研过celery和machinery这两个框架,他们都是一个异步的框架,然后这里的话我做一个简单的介绍

celery:它是一个分布式队列服务,然后它的定位更加倾向于消息流转,也就是说其实他们可以完成诸如任务的失败重试,续做这些功能,但是他不能够实现一个管理,关系化的能力,这个能力它是缺失的,而我抽象出来的这个框架更加倾向于能够进行异步任务的管理,能够对执行上下文进行修改

machinery:它是用go实现的,功能十分齐全,但是它里边的功能有些用不上,因此的话是有些重了,更为重要的是他们不支持任务的关系化管理

总而言之,他们的核心功能设计对于我们这个框架来说是收益匪浅的,但是要么就不能十分贴切我们这个框架的设计需求,因为他们都侧重于一个消息流转,没有提供一个任务执行上下文观察,任务关系化管理能力,要么就是提供了太多了能力,太重了,所以的话最终是参考了他们内部的实现,然后自己做了个轮子,在这之上DIY我们自己的功能

4. 项目是一个人做还是团队做的

这个问题就说自己一个人做的,是当时mentor给定的一个目标,将设计方案写出来之后通过了技术评审,在后期应该会参考进行实现

5. 框架要怎么接入?

框架接入实际上考察了对整个系统的熟悉程度,首先就是这个框架是依赖于MySQL和Redis进行实现的,所以在这样的情况下,需要用户提供MySQL和Redis的接入地址,然后需要创建几张表:

任务的配置表,这张表用来描述用户任务的元数据信息,比如说最大重试次数,任务类型,重试间隔等

create table t_shcedule_cgf;
insert into t_schedule_cfg ...

任务的信息表,比如说报表导出任务就是,然后我们的JobsServer就可以往里面写任务了

create table t_rpt_ouput_task

任务的位置表,这张表用来描述用户任务的要从哪几张表中去拉取,它是用来做任务治理的

当基础设施部署完毕后,就要部署JobsServer了,部署完毕后,就可以通过curl或者是http调用的方式来完成任务的创建等操作

当创建任务的设施准备完毕之后,这时候就要想办法消费任务

第一步:将任务执行的逻辑,写到worker中,这里的设计是采用了一个规范式的声明模式,也就是我们提前声明三个接口,我们的用户只需要去实现这三个接口,然后worker就可以去执行这些逻辑了,在设计中,我们是提供了这样的方法

/**
 * JobsWorker任务执行器
 */
public interface Handler {
    /**
     * 解析上下文函数
     * @param message 原始数据
     */
    void contextLoad(String message);

    /**
     * 任务执行的具体方法
     */
    void handleProcess();

    /**
     * 当本次任务执行完毕时的收尾工作
     */
    void handleFinish();

}

我们的任务处理器只需要实现这三个方法就可以了,等到完成后,就可以启动这个worker程序了

6. 框架是怎么部署的?有没有最佳实践?

实际上我们框架仅提供异步化管理任务的能力,业务方可以根据自己的需求去部署JobServer和Worker,一般都是推荐JobServer和Worker多实例部署,为什么?这是因为从设计上来说JobServer都是无状态的,所以在这样的情况下,当一个挂掉了之后,整体服务的可用性还是可以维持的

最佳实践:两台16核32G的主机在云上部署,JobServer和Worker都部署在集群中,他们都部署成3个Pod,MySQL和Redis使用主从模式,JobServer通过CLB的能力向外提供接口能力

7. Worker是托管在框架这边的服务器的吗?

不是,Worker是框架的一部分,使用Worker的时候实际上是由业务方来接入的,也就是说Worker是由业务方写完之后,自行启动起来的

框架的最佳实践是:一个业务就接入一次框架,子业务拆分成多个Worker,然后各个Worker去拉取这个业务规定的JobsServer

8. 讲一下MySQL和Redis的选型

就是说的是为什么存储不直接使用Redis?这是因为我们的需求定位中有一个关系化管理的能力,MySQL天生就是关系型数据库,因此在存储这一方面是十分方便的,同时Redis的存储并不是十分可靠的,也没有关系化的能力,因此存储最终还是选型为了MySQL

场景是任务管理场景,涉及的关系型查询很多,比如说根据任务的状态来拉取一批任务,使用MySQL的优点是数据关系清晰,存储可靠

9. 上下文如何存储的?

首先讲讲什么是上下文,所谓的上下文就是任务在执行流程中,执行环境的复现,比如说当前的执行进度,当前的处理结果,我们使用一个context字段来表示,最大是8192个字节,只有关键信息才用context存储,如果数据量大的话,推荐使用URL存储

然后实在要在上下文中存储大量数据,可以在底层引入MongoDb来存储context,目前的做法是在每一条任务的实例中添加一个context字段作为存储上下文的位置,当一个新任务被捞起来执行的时候,先执行contextLoad,然后执行handle(),最后是handleFinish()

10. 任务创建流程是怎么样的?

首先这个任务的创建流程要从外部来访问我们的接口说起,请求会打到JobsServer上,然后JobsServer解析参数完成一些基本的参数校验,然后执行一个任务的插入流程:这个过程中要涉及到几张表:

  • 任务位置表,通过读取这张表,来确定要将记录插入到哪个分表中,比如说我们的任务位置表中那两个字段begin_pos是1,end_pos是2,那么最终就会将数据插入到t_task_2中去
  • 任务信息表:插入数据的实际载体

然后最终插入成功之后,会将ID返回给调用方

11. 拉取任务接口的流程是怎么样的?

拉取任务,在迭代的前两版服务中都是由JobsServer去完成的,而且具有一定的耦合关系,主要是这样的:

首先Worker发起一个拉取任务的请求,这时候其实就是去调用HTTP/RPC请求,调用JobsServer中的holdTask接口,然后程序控制权转移到holdTask中,然后这时候会通过一个select语句去查询当前哪些任务是还没有被执行的,然后将还没有被执行的任务的基本信息拿到,返回到Worker中,然后Worker拿到这些任务就会开始执行了

但是这时候会有一个问题,就是我们的这个架构是多个Worker和多个JobsServer的,所以在这样的情况下,如果多个JobsServer同时去数据库中拉取数据的话,可能就会导致Worker占据的任务的同一批,这时候就要考虑一个并发控制的问题了

这里的话可以稍微和面试官扯一扯迭代的事情

我们刚才讲到,目前的情况是多个JobsServer同时去读取数据库导致的不同的JobsServer可能拉取到同一批任务,所以在这样的情况下,我们可以对JobsServer到数据库中执行select语句这个过程做一个限制,有两种思路:

第一种思路:直接加分布式锁,同一个时间段内只有一个JobsServer能够到数据库中去读取数据

第二种思路:同时并发到数据库中去读取任务,将并发控制交给数据库完成,我们只需要设计事务语句即可,设计出来大概是这样的

-- 1. 查询当前哪些任务还没有被捞的
select * from t_lark_task_1 where status = 1;
-- 2. 给这些任务打上一个标记
update set status = 2,owner = a where owner = '' and status = 1;
-- 3. 获取那些实际上捞到的数据
select * from t_lark_task_1 where status = 2 and owner = 'a'

这样做的好处是不需要引入别的组件,但是这样的话会导致很多无效读,比如说JobsServerA和JobsServerB,他们可能并发执行第一句的时候,可能都读取到了100000条数据,但是由于后面在并发竞争的时候,A读取到了100条,B读取到了剩余的,这样的话就会导致很多读取其实是无效的,而且数据库将会承受巨大的并发压力

因此在这两种情况下,最终采用了分布式锁的思路,然后再来考虑一个问题,就是将这个锁加在JobsServer侧,会有什么问题？

首先回顾一下拉取任务的阶段,首先是Worker发起一个HTTP调用,去调用JobsServer,如果抢到锁了,就还好,那么就直接执行了

但是如果JobsServer没有抢到锁呢?那么就意味着直接返回或者需要不断去轮询JobsServer,白白浪费了HTTP调用

转换思路:将这个锁加在Worker侧,拉取任务会变成什么样?

首先Worker发起拉取任务的请求,然后首先会去抢占一个分布式锁,如果抢占到了这个分布式锁的话就去执行,如果抢不到的话就pending

总结

将锁布置在JobsServer侧

优点

  • 从业务上容易理解,直观上解决了数据冲突访问的问题
  • 当JobsServer宕机的时候,分布式锁由于看门狗失效会自动失效,此时后续的Worker能够继续执行,避免死锁

缺点

  • 当大量的请求到来的时候,或者Worker的设计是不断轮询JobsServer的时候,会导致大量的无效请求,浪费资源

将锁布置在Worker侧

优点

  • 减少了HTTP请求调用

缺点

  • 增加了数据库的理解成本

12. Worker是手动发请求还是自动发请求拉任务?

Worker自然是通过定时任务的方式,自动发起请求去拉取任务

13. MySQL是查一大批还是查一个?

一般来说我们是批量拉取任务的,这是因为一次任务的拉取也是有开销的,如果一次只能够拉取一个任务的话就有点浪费,因此最终是拉取一批任务,至于具体拉取多少任务,是通过任务配置表来实现的,我们的任务配置表中会配置拉取一次任务会拉取多少个任务,是通过字段schedule_limit来实现的。

14. 任务的拉取是按照什么顺序?

首先和面试官拉齐这道题的认知,这个应该想问的,如果现在任务池中有10000个任务,然后本次拉取100个任务并执行,那么先拉取哪些任务呢?

这里的话聊一下方案的迭代:

第一个方案,由于任务有创建任务(create_at),用这个字段的效果就是先进先出,这种排序的方案对于短任务但是创建时间靠后的任务是不友好的,比如说,有一个长任务需要执行100s,然后它先创建,有一个短任务需要执行100ms,然后它后创建,这样的我们的短任务的执行效率就会很低

第二个方案,由于任务有更新时间(update_at),用这个字段的效果某种程度上也是先进先出,能够一定程度上解决我们这个问题,就是一个长任务被调度完了之后,它的优先级就会变低,然后就能够让出给短任务执行,更加公平,但是考虑一个问题,就是咱们的任务配置表中,为了实现自动重试,设置了一个最小的重试间隔,如果按照更新时间来排序的话就可能导致拉取到的任务达到最小的重试间隔就被拉取到了,在这样的情况下,就可能导致拉取到了大量的无效任务,拉取的条件:更新时间+重试间隔<=当前时间=>更新时间 <= 当前时间-重试间隔

select * from t_lark_task_1 where status = 1 and update_at <= '当前时间-重试间隔'

这个方案的弊端是每次都需要做一个组装,SQL不够直观

第三个方案,综合第一个和第二个方案,抽象出来一个order_time的字段,这个字段的使用方法如下:

当创建这个任务的时候,就按照它等于create_at

当调度完这个任务的时候,就按照它等于update_at

当调度这个任务失败的时候,就按照它等于update_at+max_retry_time

然后咱写SQL的话就是可以统一成:

select * from t_lark_task_1 where status = 1 and order_time <= '当前时间' order by order_time desc;

这样做的好处是:使得SQL更加容易理解,同时使得我们的重试间隔能够被用上了

引入优先级

当系统中出现这样一个需求,就是将任务分成三六九等,有一些任务可以慢点执行,有一些任务的时效性要求比较高,因此可能需要对任务的优先级做一个规划,那么我们也可以刚好在这个order_time这个字段做文章

第一个方案:引入priority字段,然后在排序的时候做联合排序即可,也就是,将优先级分为0-9,数字越小优先级越高

select * from t_lark_task_1
where status = 1 and order_time <= '当前时间'
limit 100
order by priority,order_time desc

优点

  • 直观,简单,很容易就实现了优先级功能,而且能够以优先级为主

缺点

  • 针对新引入的字段需要评估索引,如果不加索引的话就会影响效率,导致慢查询的问题,并且大多数时间,这个priority都是一样的,只有个别的任务才会设置特殊的优先级
  • priority这个字段不够灵活,比如说优先级高的任务总是会被一直调度,容易导致其他任务饥饿

order_time进一步抽象,将优先级的概念融入order_time中

优先级的目的是为了让这个任务在排序的时候能够被优先调度到,这里讲一下怎么融入

我们给priority一个含义,表示要优先多少秒执行,比如说priority = 100,就是说优先同一时间创建的普通提前100s执行

创建的时候,order_time = 现在的时间 - 100

更新的时候,order_time = 现在的时间 - 100

但是当调度失败了之后,这时候为了避免还没到时间间隔就拉取任务,因此

order_time = 现在的时间 + 最大重试间隔,避免重试间隔没用

15. 设计出来的这个优先级的使用场景是怎么样的?比如说有一个A任务和B任务,要如何定义优先级好一些?

这个问题就是说有没有实际场景的支撑,比如说音视频团队,可以定义一个超级VIP,它可以优先10小时去拉取这个任务,也可以定义一个root,直接优先一年,这样的话就可以无视其他业务

16. 这样的设计是绝对优先还是相对优先?

首先,我们的这个order_time是一个绝对的时间戳,但是我们在实现的时候,它是一个相对优先,因为绝对优先容易造成阻塞

如果在设置合理的范围内,order_time比如说10sm1min这样的数值,还是可以保证的,但是设置超过1年的时候,其实就是绝对优先了

17. 重试间隔是什么机制？

所谓的重试间隔,就是当任务执行失败了,我们想要自动重试,但是不希望立即重试,因为立即重试可能导致资源的浪费,比如说一个任务涉及到外系统的接口,然后他们在发版或者是宕机了,需要时间恢复,我们就可以暂缓这个任务的执行,等到那边大概率恢复了之后才继续执行

那么在我们的框架中,对于重试间隔这个机制采用了渐进式重试和均匀重试的方式进行,具体的实现是这样的:

在我们的任务配置元信息表中,会配置一个max_retry_interval这样的字段,表示这一类任务的默认重试间隔,然后具体到某一类任务的时候,比如说任务信息表中,还会再让用户自定义这个最大重试间隔。

然后来说说渐进式重试和均匀重试是怎么实现的

渐进式重试,它的标志是说max_retry_interval中的数字是正数,比如说10,那么渐进式重试就是[1,2,4,8,10],也就是说每次*2,当发现x2之后的数字大于了这个间隔,就等于这个间隔

均匀重试,它的标志是说max_retry_interval中的数字是负数,比如说-10,那就是每隔10s就去重试一次

还有没有其他策略?

比如说我想要渐进式的间隔自定义,要怎么实现?

可以这样,定义一个列表,比如说让用户来填[1,2,4,6,8,10]这样的数字,然后我们后端解析就可以了

18. server怎么知道任务超时了?定时器轮询?判断任务超时的方法是什么?

首先需要和面试官拉齐任务超时是什么情况:

第一种情况:一个任务被占据了之后,长时间处于一个执行中的状态,这个长时间的判断标准是由业务来设置的

这个任务超时的情况我们是由一个任务治理的模块来实现的,简单来说就是开启一个定时任务,去扫描这些超过最大执行时间的任务,最大执行时间是由业务来规定的,这类任务通常是因为一些异常的情况,worker没有上报导致的,也可能是worker挂掉的原因

19. 任务执行过程中失败了怎么办？

任务执行过程中失败了,分为两种情况

第一种情况,超过了最大重试次数,那么就会直接宣告失败,不会再回到任务池中

第二种情况,还没有超过最大重试次数,重试次数++,然后更新order_time字段,回到任务池中

20. Worker怎么竞争任务?

既然产生了竞争,那么Worker和Server就是多机部署的,这里的话经过了两个迭代

第一个迭代是说从Worker发起一个占据任务的请求,这时候会抢占一个分布式锁,只有抢占到了分布式锁才能够执行任务的占据

第二个迭代是说从将分布式锁从系统中去除,采用Kafka的方式,就是有一个同步的服务,它会定时去数据库中捞任务,然后推送到消息队列中

21. 为什么不用MySQL自带的锁

这是我们第一个方案,后面综合考虑了一下,比较不合适,当时的方案是这样的,给任务添加一个owner的字段,然后我们的一个holdtask都是和mysql打交道,sql是这样写的

-- 第一步,查询那些待执行的任务

-- 第二步,执行一个update操作,将那些待执行的任务的owner设置为当前worker的id

-- 第三步,再捞一次任务,捞那些待执行并且owner字段为当前workerid的任务

这样做的弊端有两个

  • 第一个,将并发控制权交给了mysql,性能比较低
  • 第二个,容易出现拉取任务不均匀的情况,完全取决于并发的情况,存在很多无效读的情况,比如说在第一步查询待执行的任务,查询到100条数据,但是因为并发竞争的问题,导致最后有效数据可能只有5条
  • 第三个,任务信息表并发事务多,各个业务对应的事务错综复杂,并发量大的情况下容易导致死锁

22. 用了分布式锁,就一定不会有并发问题吗？

还是会的,像Redission中的实现的这个分布式锁就有这个问题:

如果分布式锁没有一个自动续约的功能的话,如果JobsServer陷入了gc,迟迟不返回,锁过期了,其他的server就会趁虚而入了

这个框架需要任务本身自身做幂等。
