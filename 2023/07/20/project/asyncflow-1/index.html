Asyncflow项目背景

1. 了解同步与异步

微服务是一种架构的规范,在这种架构中,多个小型组件协同工作,从而构成一个整体,这种规范可以将复杂的系统分解为可管理的服务,这些服务更关注微观层面的问题,包括有单一职责,关注点分离,模块化等

那么既然将一个完整的服务拆分成了若干个微服务,那么这些服务之间就必然存在一个调用关系,服务间通信和执行流程是分布式系统的基础,它既可以是同步的也可以是异步的,这两种模式各有利弊。

同步调用

同步通信 是调用方等待响应可用的一种方式,是一个得到广泛应用的一个模式,通常来说,在目前的服务架构通常会是使用HTTP远程调用或者是RPC远程调用,每个组件都公开一个其他服务能够同步调用的接口。

程序对外部会开放一个入口,这个入口负责承接用户的流量,这个入口附近会设计一个拦截器,这个拦截器会根据流量中的元信息,将流量转发到下游服务并且等待执行,直到所有服务都已经执行完成了,每个服务可以顺序或者并行地去执行规定好的下游服务。

一个调度器可以显式地去调度所有的服务(通过直接发起同步调用的方式),也可以通过异步任务的方式,将一个请求暂存在某个中间件中,下游服务定时读取请求内容,之后隐式地完成即可。

去中心化和同步

去中心化的同步通信方式在入口点拦截流量,拦截器将请求转发到下一步并且等待响应,这个循环将会一直下行,直到所有的请求都被执行完毕,每个服务可以串行地或者并行地去调用其他服务,这种实现起来比较简单,但是流程细节分布在整个系统之中,从而导致组件之间的耦合。

这种方式适用于上下游服务的读写频率对等的时候,这时候可以快速响应消费者的请求,但是当读写频率不对等的时候,这时候调用链路中的任何一个服务都可以会导致整条调用链路的阻塞,从而降低QPS,甚至更加严重的是,丢失用户的请求

这种方式对于需要经常发生变更的工作流的来说是不合适的(指的是TPS不稳定,或者是需要在工作流上插入新的服务的工作流都不是特别适用)

去中心化的特点就是每个服务都具有自己的权重

编排、同步和顺序

这种模式并不是一种去中心化的方式,它引入了一个角色中央编排器,编排器依然是拦截服务,它处理传入的流量并且将其转发到下游服务上,并且接收来自下游服务器的响应,它的责任可以概述为接收来自外界的请求,对下游服务不断调用,并且对服务返回的结果进行决策,这使得这种设计的系统不是一个去中心化的系统,编排器起到了中央管理器的作用,一旦中央编排器挂掉,那么整个系统就无法运行了,哪怕真正处理请求的服务都没有挂掉,它适合那种读取多的系统,因为通常来说读取的系统希望能够得到更快的响应速度,而中央编排器可以根据各个服务的响应来确定本次响应是否成功,去中心化的系统将调用权下放到了各个服务,这也就意味着下游服务之间的调用响应判断更加复杂。

队列可以缓解上游服务的激增,一个很重要的理念就是削峰填谷,它的一个核心理念就是:当上游服务生产速率很高,而下游服务消费速率很低的时候,队列可以作为一个中介,这个中介可以避免下游服务在短时间内承受过多流量而被打挂,如果做不到这点,可能会出现级联故障,一种策略是使用一种断路器的组件,这样的话就可以缓解同步系统中的流量激增

增加负载均衡和服务发现的开销:可以通过引入负载均衡来解决流量激增的问题,主要就是将激增的流量分散到相同服务的不同副本中去,这样做可以缓解流量激增的问题,但是也增大了服务集群间的维护成本和各服务之间要引入一个服务的注册中心来实现服务发现的的能力,这样的话也造成了一定成本

产生耦合:同步系统内部会产生耦合,服务之间没有抽象,服务之间直接调用,服务地址写死在服务中,产生了强烈的关联,增加了系统的复杂性

异步调用

异步通信非常适合分布式体系架构,不需要等待响应,就可以将一个服务和多个服务分开,异步通信的实现方式通常有几种方式RPC、或者消息引擎

在这种模式中,每个组件都监听总线的消息,事件的到来是信号,执行所需要的上下文是事件的负载的一部分,触发下游事件是每个服务所拥有的责任

这种方式遵循工作分散的原则,不存在服务之间的直接调用,但是当需要同步调用的时候,这时候还是要进行协调,避免上游服务饥饿

异步系统的一些权衡

中间的消息引擎需要提供高可用,一旦消息引擎产生故障,那么整个异步系统就无法工作了

同步包装:提供一个调用下游异步服务的入口点,并且在本服务中保存请求状态,只有当下游服务处理完成后,才返回结果,这将带来一定的延时,并且占用一定的资源

2. 需求背景与项目场景

在一些项目中,对外的接口除了以同步接口的形式提供,很多业务还会涉及到一个异步的流程,比如说音视频的处理,区块链,审核等耗时操作,比如说在实习中,对于一个大文件的导出,并不是同步的,而是发起一个异步任务,异步流程通常是这样的玩法:

  • 外部调用同步接口,生产一个任务
  • 在后台异步处理任务
  • 发起者轮询任务

Asyncflow要做就是为这一类的任务开发一个框架,通常需要考虑一个任务的管理,异常处理,流量控制等细节,通常来说开发需要两周,但是用这个框架一天就能够完成

问题:这个任务为什么不能够直接用消息队列完成?

原因:这是我们的场景决定的,第一,我们要支持一个上下文的更新和传递操作,异步场景通常来说会将一个大任务拆分成一个个的小任务,比如说一个视频的处理任务,第一个阶段是查询视频的元数据,第二个阶段是鉴黄,第三个阶段是转码,每一个阶段的结果都需要我们去记录下来,不然丢失了消息的话就又是资源的浪费,但是这个问题可以通过消息流程模型来解决,解决的思路是这样的,当每一个消息产生之后,这时候就将这个消息投入到下一个topic中去,不断流转

第二,核心问题还是在于这类异步场景的任务需要一个任务管理数据看板,比如说某个用户下进行的任务列表,这些都是常见的查询方式

select * from task where user_id = "xxx"

而如果涉及到这个需求,那么就必须提供关系化任务的能力了,因此如果单纯只是要完成任务的调度的话,实际上使用消息的调度模型就可以解决这个问题了

适用场景

一个任务需要被拆分为多个阶段,阶段之间是同步的,比如说一个任务被拆分成子任务1,子任务2,子任务3,子任务2只有在等待子任务1完成后才能继续运行,这个就很像责任链模式,在实习的时候责任链的设计也是用到了一个context来传递必要的参数

  • 场景1:音视频:审核(鉴黄)=>转码=>上传

任务上下文:就是按照需求定义的对象结构,无所谓是什么,只要有需要的信息就可以了,存储模型中规定,将上下文序列化为一个json格式的消息,然后存到mysql中作为一个任务阶段

3. 项目功能需求列表

功能列表

  • 支持任务创建(同步返回任务的ID)
  • 支持通过任务ID查询
  • 任务自动调度,维护任务状态
  • 支持各种任务快速注册
  • 任务支持优先级调度
  • 服务挂掉之后可以自动续做
  • 灵活的任务配置
  • 任务配置支持持久化

性能指标

2核4GB的虚拟机下,创建任务接口,占据任务接口,查询任务接口能够达到2000QP,支持水平扩展
