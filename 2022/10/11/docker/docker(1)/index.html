后端开发基础

1. Docker基本架构

1.1 Docker基本概念

什么叫容器?

一句话来概括容器,就是说将软件打包成标准化单元,可以用于开发、交付和部署。容器镜像是轻量级的,可执行的独立软件包,包含软件运行所需要的所有内容:包括有代码,运行时环境,系统工具,系统库和设置

容器化软件适用于基于Linux和Windows的应用,在任何环境中都能够始终一如地运行

容器赋予了软件独立性,使其免受外在环境差异,例如开发和预演环境的差异,从而有助于团队间在相同基础设施上运行不同的软件时的冲突

简单来说,容器就好像一个存放东西的地方,这个地方根据开发者的需要,集成了运行软件,运行环境等等。

  • 镜像(Image):Docker镜像(Image)就相当于是一个root的文件系统。
  • 容器(Container):镜像和容器的的理解就类似于Java中的类与对象的关系一样，镜像是静态的定义，容器则是在运行时的实例。容器可以被创建、启动、停止、删除、暂停等
  • 仓库(Repository):仓库可以看作是一个仓库代码控制中心
  • Docker客户端(client):Docker客户端通过命令行或者其他命令工具使用Docker SDK与Docker的守护进程通信
  • Docker Registry仓库用来保存镜像，可以理解为代码控制中的代码仓库。一个Docker Registry可以包含多个仓库(Repository)，每个仓库可以包含多个标签(Tag)，每个标签对应一个镜像。通常，一个仓库会包含同一个软件不同版本的镜像，可以用**<仓库名>:<标签>**的格式来指定具体这个软件哪个版本的镜像，如果不给出标签，将以latest作为默认的标签。

Docker使用客户端-服务器(C/S)架构模式，使用远程API管理和创建Docker容器

物理机和虚拟机有什么区别?虚拟机和容器又有什么区别?

首先对于物理机和物理机之间,他们的相互隔离的资源应该说是硬件环境,他们具有不同的CPU、硬盘等

对于物理机和虚拟机之间,实际上虚拟机是借助于物理机上的硬件模拟出一套自己的硬件,然后再运行一套操作系统。

虚拟机和容器的区别,主要集中在隔离级别上,虚拟机之间的操作系统资源是互不共享的,比如说内存资源,CPU资源等

因为他们运行起来了一套完整的操作系统,而容器则是共享同一套操作系统资源,容器的隔离级别会稍低一些

Docker容器通过Docker镜像来创建

Docker容器有什么特点?

我认为Docker容器的安全性主要体现在:

  • 它是轻量级的:通过文件系统层进行构造镜像,共享一些文件,尽量降低磁盘的使用量,从而能够更快地下载镜像
  • 标准化的:就是说在当前的主流的Linux的系统下都可以运行Docker,兼容性很好
  • 安全的:容器之间是相互隔离的,即使是有一台容器出现了问题,也不会导致宿主机宕机

为什么要用Docker?

正如它的特点,它首先是轻量级的,部署一个应用或者特定的系统只需要几百兆的磁盘空间,内存占用低,可以在同一个宿主机上运行很多个这样的应用。

解决了环境部署繁杂的问题,Docker的镜像提供了除了内核完整的运行环境,确保了应用运行环境的一致性,从而不会出现:这段代码在某个机器上是没有问题的,但是换了个机器有问题了

当需要使用公有的服务器跑代码的时候,避免某一个用户的操作不当而导致服务器宕机

迁移方便:当有大批量的容器需要移动的时候,可以很快地迁移,不需要担心运行环境的变化而导致应用无法运行

使用Docker可以定制应用镜像来实现持续CI/CD

1.2 说说Docker的生命周期

1.2.1 镜像是什么?如何构建的?

操作系统分为内核空间和用户空间,对于Linux而言,内核在启动之后,会挂载root文件系统为其提供用户空间支持,而Docker镜像,就相当于一个root文件系统

Docker镜像是一个特殊的文件系统,除了提供容器运行时所需要的程序、库、资源、配置等文件外,还包含了一些为运行时准备的一些配置参数,镜像不包含任何动态数据,其内容在构建之后也不会被改变

Docker 设计时，就充分利用 Union FS 的技术，将其设计为分层存储的架构 。镜像实际是由多层文件系统联合组成。

镜像的构建过程是这样的:它会一层层的构建,前一层是后一层的基础,每一层在构建完毕之后就不会再发生改变,后一层上的任何改变只发生在自己这一层,比如说删除前一层文件的操作,实际上并不是针对删除前一层文件,而只是在当前层标记这个文件已经删除。在最终容器运行的时候,虽然不会看到这个文件,但是实际上这个文件会一直跟随着镜像

每一层尽量只包含该层所需要添加的东西,任何额外的东西应该要在该层构建结束之前被清理掉

分层存储的特征还使得镜像的复用,定制变得更加容易

1.2.2 容器是什么?如何构建的?

容器的实质是进程，但与直接在宿主执行的进程不同，容器进程运行于属于自己的独立的 命名空间。前面讲过镜像使用的是分层存储，容器也是如此。

容器存储层的生存周期和容器一样，容器消亡时，容器存储层也随之消亡。因此，任何保存于容器存储层的信息都会随容器删除而丢失。

按照 Docker 最佳实践的要求，容器不应该向其存储层内写入任何数据 ，容器存储层要保持无状态化。所有的文件写入操作，都应该使用数据卷（Volume）、或者绑定宿主目录，在这些位置的读写会跳过容器存储层，直接对宿主(或网络存储)发生读写，其性能和稳定性更高。数据卷的生存周期独立于容器，容器消亡，数据卷不会消亡。因此， 使用数据卷后，容器可以随意删除、重新 run ，数据却不会丢失。

2.Docker- HelloWorld

2.1 输出hello world

Docker允许你在容器内运行应用程序，使用docker run命令在容器内运行一个应用程序。

  • 输出hello world
docker run ubuntu:15.10 /bin/echo "hello world"

参数解释

  • docker:指令执行前缀,具体指代的是Docker二进制执行文件
  • run:与前面的docker组合起来运行一个容器
  • ubuntu:15.10:指定要运行的镜像，Docker首先从本地主机上查找镜像是否存在，如果不存在，Docker就会去镜像仓库Docker Hub下载公共的镜像。
  • /bin/echo:在容器中要执行的命令

2.2 运行交互式的容器

可以通过docker的两个参数,-i和-t来让docker运行的容器具有对话的能力

docker run -i -t ubuntu:15.10

参数解释

  • -t:在新容器中指定一个伪终端或者终端
  • -i:允许你对容器内的标准输入stdin进行交互

​ 可以通过exit或者CTRL+D来退出终止容器

2.3 启动容器(后台模式)

可以使用以下命令创建一个以进程方式运行的容器

docker run -d ubuntu:15.10 /bin/sh -c "while true;do echo hello world;sleep1;done"
6e7264cbacd2d8cdc097769bee6dda6cf233fafee33ae117e5e01705d5c759d4

参数解释

/bin/sh:表示使用乌班图系统中的脚本解析工具执行脚本文件。

其中第二行这一大串表示的是容器的ID，这对于每个容器来说都是唯一的，我们可以通过容器ID来知道对应的容器发生了什么

查看容器的情况的命令是:docker ps -a:-a代表的是列出所有的容器的情况，如果没有-a，那么就只会列出运行中的容器。

观察其参数列表

CONTAINER ID   IMAGE          COMMAND                  CREATED         STATUS         PORTS     NAMES
  • CONTAINER:容器名称
  • IMAGE:代表该容器是由哪个镜像实例化而来
  • COMMAND:启动容器时执行的命令
  • CREATED:容器的创建时间
  • STATUS:容器的状态
    • created:已经创建
    • restarting:重启中
    • running 或者Up:正在运行
    • removing:迁移中
    • paused:暂停
    • exited:停止
    • dead:死亡
  • Port:容器的端口信息和使用的连接类型tcp/udp
  • NAMES:自动分配的容器名称

在宿主机中使用docker logs ID/name命令就可以查看容器内的标准输出

可以通过docker stop ID/Name来停止容器

3. docker容器的使用

3.1 基础指令

  • docker pull image:载入镜像，如果本地没有则会从远程仓库中下载

  • docker run -it image /bin/bash:指定以-it(交互式终端)启动容器

  • docker start/restart:启动一个已停止的容器

  • docker run -itd --name ubuntu-test ubuntu:15.10 /bin/bash

在大部分的场景下，我们希望docker服务是在后台运行的，因此可以使用-d来指定容器的运行方式

  • docker stop:停止一个已经启动的容器

  • docker attach:可以进入一个容器，但是退出容器的时候会导致容器的停止

  • docker exec -it CONTAINER_NAME /bin/bash:可以进入一个容器，指定以某种方式例如/bin/bash，退出的时候不会导致容器停止。

  • docker export:如果要导出某个容器，可以使用该指令，例如

docker export CONTAINER_ID > exporter.tar
  • docker import:导入容器快照
cat exporter.tar | docker import - test/ubuntu:v1

将该快照文件ubuntu.tar导入到镜像，此外，也可以通过指定URL或者某个目录来导入

  • docker rm -f CONTAINER_ID:删除容器
  • docker container prune:清除掉所有处于终止状态的容器

3.2 网络映射相关指令

运行一个web应用

docker pull traning/webapps
docker run -d -P traning/webapp python app.py

参数解释

  • -d:让容器在后台运行
  • -P:将容器内部使用的网络端口随机映射到我们使用的主机上

查看Web应用

runoob@runoob:~#  docker ps
CONTAINER ID        IMAGE               COMMAND             ...        PORTS                 
d3d5e39ed9d3        training/webapp     "python app.py"     ...        0.0.0.0:32769->5000/tcp

参数解释:最重要的是PORTS这个参数，其结构是

0.0.0.0:32769->5000/tcp:首先0.0.0.0指代的是宿主机的IP，由于docker并不知道也并不关心，因此使用0占位符，其后接的32769表示的主机的端口地址，而->后的5000表示的是docker虚拟机中的5000端口

同时可以通过-p这个参数来设置不一样的端口

docker run -d -p 8080:8080 traning/webapp python app.py

网络端口的快捷方式

docker port可以指定ID或者(名字)容器的某个端口映射到宿主机的端口号

查看WEB应用程序日志

docker logs -f CONTAINER_ID

查看WEB应用程序容器的进程

docker top CONTAINER_ID/NAME

检查WEB应用程序

docker inspect:可以用来查看docker的底层信息。它将会返回一个JSOn文件记录着Docker容器的配置和状态信息。

4. Docker镜像使用

当运行容器的时候，如果其使用的镜像在本地是不存在的话，docker就会自动从docker镜像仓库中去下载，默认是从docker hub公共镜像源下载

4.1 管理和使用本地Docker主机镜像

  • docker images
REPOSITORY          TAG                 IMAGE ID            CREATED             SIZE
  • REPOSITORY:表示镜像的仓库源
  • TAG:镜像的标签
  • IMAGE ID:镜像的ID
  • CREATED:镜像创建的时间
  • SIZE:镜像的大小

注:同一仓库源可以有多个TAG，代表这个仓库的不同个版本，如果你不指定一个镜像的版本标签，将默认使用latest

查找镜像

可以使用docker search来搜索镜像，例如docker search httpd

NAME DESCRIPTION STARS OFFICIAL AUTOMATED
  • NAME：镜像仓库源的名称
  • DESCRIPTION:镜像的描述
  • STARS:点赞数
  • OFFICIAL:是否官方
  • AUTOMATED:自动构建

删除镜像

docker rmi imges:删除镜像

4.2 创建镜像

当我们从docker镜像仓库中下载的镜像不能满足我们的需求的时候，可以通过以下两种方式来对镜像进行修改

  • 从已经创建的容器中更新镜像，并且提交这个镜像
  • 使用Dockerfile指令来创建一个新的镜像

更新镜像

更新镜像之前，我们需要使用镜像来创建一个容器,以乌班图操作为例:

docker run -i -t ubuntu:15.10 /bin/bash

等容器跑起来之后，执行变更代码

apt-get update

此时容器内的内容与其实的镜像状态已经不同，此时我们将这时候的容器保存为新的镜像

docker commit -m="update" -a="Lumxi" cb84304d2da7 lumxi/ubuntu:v0.1

参数解释

  • -m:输入提交信息
  • -a:输入镜像的创造者
  • cb84304d2da7:容器名称
  • lumxi/ubuntu:v0.1:镜像名称

然后可以使用该镜像来构造容器

docker run -it lumxi/ubuntu:v0.1 /bin/bash

4.3 构建镜像

可以使用docker build，从零开始来创建一个新的镜像，为此，我们需要创建一个Dockerfile文件，它包含了一组指令来告诉docker如何来构建我们的镜像

[root@localhost ~]# cat Dockerfile 
FROM    centos:6.7
MAINTAINER      Fisher "fisher@sudops.com"

RUN     /bin/echo 'root:123456' |chpasswd
RUN     useradd runoob
RUN     /bin/echo 'runoob:123456' |chpasswd
RUN     /bin/echo -e "LANG=\"en_US.UTF-8\"" >/etc/default/local
EXPOSE  22
EXPOSE  80
CMD     /usr/sbin/sshd -D

每一条指令都会在镜像上创建一个新的层，每一个指令的前缀都是大写的。

第一条FROM指定了使用哪个镜像源

RUN指令告诉docker在镜像内要执行命令，安装了什么

然后我们使用Dockerfile文件，通过docker build命令来构造一个镜像

docker build -t lumxi/centos:6.7 .

-t:指定要构建的镜像名

.:表示Dockerfile所在的目录

设置镜像标签

docker tag CONTAINER_ID IMAGES_TAG

5. Docker容器连接

之前的学习中通过网络端口可以用来访问运行在docker容器内的服务，容器中可以运行一些网络应用，要让外部也可以访问这些应用，可以通过-p参数来指定端口映射，格式为本地端口:容器端口

5.1 网络端口映射

除了使用-p标识来指定容器端口绑定在主机端口，还可以绑定其网络地址，例如:

docker run -d -p 127.0.0.1:8080:8080 training/webapp python app.py

这样做的话，就可以通过访问127.0.0.1:8080来访问容器的8080端口

如果要绑定UDP端口，那么可以在端口后面加上/udp，例如

docker run -d -p 127.0.0.1:8080:8080/udp training/webapp python app.py

5.2 容器互联

端口映射并不是唯一把docker连接到另外一个容器的方法，docker有一个连接系统允许将多个容器连接在一起，共享信息。docker连接会创建一个父子关系，其中父容器可以看到子容器的信息

容器命名

当我们创建一个容器的时候，docker会自动对其进行命名，另外也可以使用--name表示来命名容器。

新建网络

docker network create -d bridge test-net
  • -d参数指定:选择有bridge模式和overlay模式

容器连接与绑定

运行 两个新的容器并且连接到新的网络

docker run -itd --name test1 --network test-net ubuntu:15.10 /bin/bash
docker run -itd --name test2 --network test-net ubuntu:15.10 /bin/bash

测试两台机器的连通性，发现正常。

手动指定容器的配置

docker run -it --rm -h host_ubuntu  --dns=114.114.114.114 --dns-search=test.com ubuntu

–rm：容器退出时自动清理容器内部的文件系统。

-h HOSTNAME 或者 –hostname=HOSTNAME： 设定容器的主机名，它会被写到容器内的 /etc/hostname 和 /etc/hosts。

–dns=IP_ADDRESS： 添加 DNS 服务器到容器的 /etc/resolv.conf 中，让容器用这个服务器来解析所有不在 /etc/hosts 中的主机名。

–dns-search=DOMAIN： 设定容器的搜索域，当设定搜索域为 .example.com 时，在搜索一个名为 host 的主机时，DNS 不仅搜索 host，还会搜索 host.example.com。

6. 容器与虚拟机的对比

首先先从容器和虚拟机的概念说起吧

容器它是一种软件应用的抽象,它可以理解成核心软件应用+运行环境的总和。

虚拟机它是一种物理硬件的抽象,它可以理解成对物理机硬件设备的抽象,通过这些抽象,就可以把一部分资源分配到对应的虚拟机上,而虚拟机就可以利用这些资源,为自己生成操作系统,看起来,就好像一台服务器上同时又运行了多个服务器一样。

至于对比的话:

  • 容器的运行逻辑是这样的,多个容器可以在同一台机器上运行,共享系统内核,但是各自作为独立的进程在用户空间中运行,与虚拟机相比,容器所占用的空间是比较少的,容器的镜像大小通常只有几十兆而已
  • 虚拟机的运行逻辑是这样的,管理程序允许多个VM在同一台机器上运行,每个VM都包含了一整套的操作系统,一个或者多个应用,必要的二进制文件和库资源,因此占用了大量的空间,启动缓慢
